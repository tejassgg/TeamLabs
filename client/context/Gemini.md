# 🤖 TeamLabs - Gemini AI & RAG System

This document outlines the architecture, model selection, configuration settings, and data synchronization workflows for the Gemini-powered AI analytics and Retrieval-Augmented Generation (RAG) system in TeamLabs.

---

## 🛠️ SDK & Initialization

TeamLabs leverages Google's latest Generative AI SDK:
*   **Package**: `@google/genai` (version `^2.10.0`).
*   **Initialization**: The service imports the SDK using standard CommonJS modules and instantiates the client client-side and server-side:
    ```javascript
    const { GoogleGenAI } = require('@google/genai');
    this.geminiAI = new GoogleGenAI({}); // Automatically resolves process.env.GEMINI_API_KEY
    ```

## 🔮 Model Configurations

The platform utilizes two distinct models optimized for performance and cost-effectiveness:

### 1. Vector Embeddings: `gemini-embedding-001`
*   **Purpose**: Vectorizes text chunks to support semantic search.
*   **Output Dimensionality**: Configured explicitly to **768 dimensions** for consistency across the document store.
*   **Trigger**: Invoked whenever project resources (tasks, activities, comments) are updated or when a user issues an AI search query.

### 2. Text Generation: `gemini-3.5-flash`
*   **Purpose**: Summarizes project telemetry, tracks status patterns, and writes executive, technical, or standard progress reports.
*   **Optimization**: Configured with a `thinkingBudget: 0` inside the generation config. This disables internal thinking steps to return response tokens as fast as possible for web dashboards.
    ```javascript
    const response = await this.geminiAI.models.generateContent({
      model: "gemini-3.5-flash",
      contents: prompt,
      config: {
        thinkingConfig: {
          thinkingBudget: 0,
        },
      }
    });
    ```

## 📚 Retrieval-Augmented Generation (RAG) Flow

To prevent LLM hallucination and ensure reports are grounded in actual project telemetry, the system implements a custom RAG flow:

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  Data Sync   │────>│ Text Chunking│────>│  Embedding   │
│  (Cron/User) │     │ (Sentence pt)│     │  Vector Gen  │
└──────────────┘     └──────────────┘     └──────────────┘
                                                 │
                                                 ▼
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│ Prompt + RAG │<────│ Cosine Sim   │<────│ MongoDB      │
│ Context Gen  │     │ (>0.7 Match) │     │ Vector Store │
└──────────────┘     └──────────────┘     └──────────────┘
```

### 1. The Chunking Pipeline
Large texts (like project briefs, lengthy comments, or meeting logs) are processed using an overlapping window approach in [embeddingService.js](file:///server/services/embeddingService.js):
*   **Max Chunk Size**: 800 characters (for knowledge base index chunks) up to 1000 characters.
*   **Overlap Size**: 100 characters to prevent loss of context across windows.
*   **Sentence Preservation**: The system attempts to slice chunks at sentence boundaries (searching back for punctuation markers like `.`, `!`, or `?`) to preserve readability.

### 2. MongoDB Vector Store (`KnowledgeBase` Model)
Chunks are stored in MongoDB using the [KnowledgeBase.js](file:///server/models/KnowledgeBase.js) schema:
*   **`embedding`**: Array of 768 floats generated by `gemini-embedding-001`.
*   **`sourceType`**: Specifies source collection (`project`, `task`, `report`, `user_activity`, `team`, `comment`, `attachment`).
*   **`sourceId`**: Reference ObjectID of the original database document.
*   **`organizationId`**: Boundaries search query constraints within tenant limits.

### 3. Semantic Cosine Similarity
Vector retrieval uses a custom JavaScript implementation of Cosine Similarity (dot product normalized by Euclidean norms):
*   **Equation**:
    $$\text{Similarity}(A, B) = \frac{A \cdot B}{\|A\| \|B\|}$$
*   **Threshold**: Set to `0.7` by default (or `0.6` in fallback query controllers).
*   **Limit**: Caps matching segments at a maximum of `10` results.
*   **Fallback Sync**: If a query returns 0 matches, the system triggers a background sync to fetch recent records, vectorize them, and try the search again.

## 📂 Data Syncing Service

The [knowledgeBaseSyncService.js](file:///server/services/knowledgeBaseSyncService.js) manages vector database updates:
*   **Batch Processing**: Indexes items in batches of 10 with a 1-second delay between batches to respect Gemini API rate limits.
*   **Metadata Extractor**: Maps entities to structured texts. For example, a project is summarized with its Name, Description, Owner, Github Link, Stars, and Forks before chunking.
*   **Clean and Rebuild API**: Supports clearing an organization's embeddings (`KnowledgeBase.deleteMany({ organizationId })`) and conducting a clean rebuild (`regenerateEmbeddings`) to fix indexing anomalies.

## 💬 AI Chatbot API

The chatbot interface is routed via [chatbotRoutes.js](file:///server/routes/chatbotRoutes.js):
*   **Predefined Keyword Routing**: Intercepts common navigation intents (such as asking "how to register," "how to open kanban," "what is a project") to instantly return route URLs and action buttons.
*   **Fallback State**: Falls back to support routing when requests are outside the system scope, with historical conversations logged inside the `ChatConversation` schema.
